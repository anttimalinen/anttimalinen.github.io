## Description

A firewall answers one question: should this connection be allowed at all. It does not look at what rides inside an allowed connection, which is exactly the gap Snort exists to close. Snort is an open-source, rule-based Network Intrusion Detection and Prevention System (NIDS/NIPS), originally written by Martin Roesch and now maintained by Cisco's Talos team since Cisco's 2013 acquisition of Sourcefire. Cisco's own description calls it the foremost open-source IPS in the world: a tool that inspects live or captured traffic against a ruleset and generates alerts, or drops packets outright, when something matches.

This post covers Snort's four operating modes, from basic packet sniffing through full intrusion detection and prevention, then works through writing detection rules from the ground up. Every command, flag, and answer below comes from hands-on work against the room's lab machine and its provided pcap files.

## The problem

Network traffic at any real scale is too much for a person to watch packet by packet. A single mid-sized office generates thousands of connections an hour; a single malicious payload might be one packet among millions. Something has to do the watching automatically, compare what it sees against known-bad patterns, and only surface what actually matters. That is the entire premise of an IDS: passive detection at scale, built on rules that encode what "bad" looks like at the packet level.

The distinction between detection and prevention is not academic. An IDS raises a hand and says something is wrong; a human or a downstream system decides what happens next. An IPS sits in the traffic path directly and makes that call itself, dropping or rejecting the packet before it ever reaches its destination. Choosing between the two is a real operational tradeoff, not a strictly-better-or-worse choice. An IPS blocking a false positive breaks legitimate traffic in real time, while an IDS flagging the same false positive just wastes an analyst's few minutes reviewing it. That risk calculus is why so many organizations run new rules in detection-only mode before ever promoting them to blocking mode, and why Snort exposes both behaviors as configurable choices rather than picking one for you.

## How it works

**Step 1: IDS vs IPS fundamentals**

Before touching Snort itself, the room lays out the taxonomy that everything else builds on. Two root categories, IDS (detect) and IPS (prevent), each splitting further by where they watch.

| Type | Detects or prevents | Scope |
|---|---|---|
| NIDS (Network IDS) | Detect | Entire subnet |
| HIDS (Host IDS) | Detect | Single endpoint |
| NIPS (Network IPS) | Prevent | Entire subnet |
| HIPS (Host IPS) | Prevent | Single endpoint |
| WIPS (Wireless IPS) | Prevent | Wireless traffic specifically |
| NBA (Network Behavior Analysis) | Prevent | Entire subnet, anomaly-based |

Matching a control to a scenario comes down to two questions: is the goal to stop the threat or just see it, and does the answer need to cover a whole network or one machine. Stopping a threat on a single machine is HIPS. Detecting on a full network is NIDS. Detecting on a single machine is HIDS. Stopping across a full network is NIPS.

NBA deserves a separate note because it is not really a fifth category so much as a different detection philosophy layered onto the same NIPS scope. Per Cisco's own description, Snort's core identity is as a full-blown network intrusion prevention system, and NBA works similarly to NIPS in that both protect an entire subnet and both terminate malicious connections. The difference is what triggers that termination. Standard NIPS matches traffic against known signatures. NBA instead requires a training period, known as baselining, during which the system learns what normal traffic on that specific network looks like before it can reliably flag what is abnormal. That baselining period is also the system's biggest operational risk: if an attacker is already active on the network while the baseline is being established, that malicious traffic gets learned as normal and never gets flagged again.

Underneath deployment scope sits a second axis: how detection actually happens.

| Technique | Approach | Best at |
|---|---|---|
| Signature-based | Matches known patterns of malicious behavior | Known threats |
| Behavior-based | Compares observed activity against learned normal behavior | New, unknown threats |
| Policy-based | Compares activity against configured security policy | Policy violations |

Snort operates primarily as a signature-based engine: rules encode a specific pattern, and traffic either matches or does not. That gives Snort speed and low false-positive rates against known threats, at the direct cost of being blind to anything that has never been written into a rule yet, which is exactly the gap behavior-based detection like NBA exists to cover.

**Step 2: First interaction with Snort**

Before running any real analysis, verifying the instance and its configuration is the same first move any tool deserves. Running `snort -V` returns the version banner, including the build number. This lab's instance reports build 149.

Configuration files control everything Snort does beyond raw sniffing: which rules load, which preprocessors run, how output gets formatted. Testing a configuration file without actually processing traffic uses `-T` alongside `-c` to point at the file, run as `sudo snort -c /etc/snort/snort.conf -T`. Scrolling to the rule count summary at the end of that output shows 4151 rules loaded from the default snort.conf. Pointing the same test at a second configuration file with `sudo snort -c /etc/snort/snortv2.conf -T` tells a different story: only 1 rule loads.

That gap between 4151 and 1 is not a bug; it demonstrates the entire point of separating configuration files. A minimal configuration with a single custom rule is exactly what you want when testing a new detection idea in isolation, without thousands of unrelated default rules generating noise in the output alongside it. Running the wrong config file against the wrong task is a real, easy mistake, which is why checking the loaded rule count first, before ever touching a pcap, catches that mistake before it wastes time.

**Step 3: Sniffer mode**

Sniffer mode is Snort at its simplest: read packets, print them, nothing logged. Five flags control how much detail gets printed.

| Flag | Shows |
|---|---|
| -v | Verbose TCP/IP summary in the console |
| -d | Packet payload data |
| -e | Link-layer (Ethernet/TCP/IP/UDP/ICMP) headers |
| -X | Full packet detail in hex and ASCII |
| -i | Select a specific network interface |

These stack in a clear escalation. `-v` alone gives a tcpdump-style one-line summary per packet: source, destination, protocol, length. Adding `-d` layers the actual payload bytes underneath each summary, useful the moment a summary line alone cannot answer what a packet actually contained. Adding `-e` on top of that surfaces the Ethernet-layer MAC addresses and frame type, useful when the question is about the local network segment rather than the IP-layer conversation. `-X` skips the incremental approach entirely and dumps everything, full hex alongside the ASCII interpretation of every byte in the packet, header through payload, which is the mode to reach for when nothing less detailed answers the question. `-i` sits apart from the rest: it does not add detail, it selects which network interface Snort listens on, defaulting to the only available interface when just one exists.

**Step 4: Packet logger mode**

Sniffing shows traffic live. Logger mode captures it to disk for later analysis, which matters the moment an investigation needs to happen after the fact rather than while packets are actively flowing.

| Flag | Function |
|---|---|
| -l | Set the log/alert output directory (default /var/log/snort) |
| -K ASCII | Log packets in human-readable ASCII format instead of binary |
| -r | Read back a previously logged file |
| -n | Limit processing to a specific number of packets |

One operational detail worth knowing before running any of this: Snort needs root privileges to sniff traffic, so every file it logs gets created under the root account. Reading those logs back later needs either continued root access or a manual ownership change with `sudo chown` to hand the files to a regular user account. It is a small detail, but it is the kind of thing that stops a fresh analyst cold the first time they try to read a log file and get a permissions error.

The default binary log format is efficient but not human-readable without Snort, tcpdump, or Wireshark to parse it. `-K ASCII` trades that efficiency for direct readability: instead of one binary blob, it produces a folder structure organized by source IP, with individual files named after the connection they represent, in a sourceport-destport format like 3009-53.

That naming convention is exactly how this room's hands-on exercise gets solved. Running the exercise's traffic generator and logging in ASCII mode with `sudo snort -dev -K ASCII -l .` against the target folder 145.254.160.237 surfaces a filename revealing the source port used to connect to port 53: 3009.

Switching to reading a previously captured binary log, snort.log.1640048004, with `snort -r snort.log.1640048004 -n 10` limits output to the first ten packets. Reading the IP ID field on the 10th packet returns 49313. Repeating the same read with `-X` added, to surface full packet payload in hex and ASCII, and inspecting the fourth packet's payload for an HTTP Referer header returns http://www.ethereal.com/development.html. Reading the eighth packet's TCP header for its acknowledgment number returns 0x38AFFFF3. Filtering the same log with a Berkeley Packet Filter expression for TCP traffic on port 80 and reading the summary count at the end returns 41 matching packets.

Five different questions, one log file, a handful of flag combinations. That progression is the actual skill this task teaches: knowing which flag surfaces which layer of a packet, rather than dumping full `-X` output every time and manually scanning for the answer.

**Step 5: IDS/IPS mode**

Sniffing and logging are passive. IDS/IPS mode is where Snort starts actively evaluating traffic against rules and deciding what to do about a match.

| Flag | Function |
|---|---|
| -c | Load a configuration file |
| -T | Test the configuration without processing traffic |
| -N | Disable logging (console output still works if separately enabled) |
| -D | Run as a background daemon |
| -A | Set the alert output mode |

The `-A` flag carries five distinct sub-modes, and picking the wrong one for a given task wastes real time: full (the default, every possible detail about a match), fast (message, timestamp, source and destination, no payload detail), console (fast-style output printed live to the terminal instead of a file), cmg (basic header info plus payload in hex and text, also printed live), and none (disable alerting entirely, while still logging traffic in binary format). Only console and cmg produce anything visible in the terminal while Snort runs; the other three write silently to disk and require reading the alert file afterward to see anything at all, which is an easy trap for anyone expecting live feedback and getting a blank terminal instead.

Running the default configuration against this task's generated traffic with `sudo snort -c /etc/snort/snort.conf -A full -l .` and reading the resulting HTTP summary after the traffic generator finishes and Snort is stopped shows 2 detected HTTP GET requests.

One detail from this task's reading worth carrying forward: everything above runs Snort as an IDS, alerting on matches but never blocking anything. Actually dropping traffic requires inline mode, activated with `-Q --daq afpacket` and at least two network interfaces bridged together. The room's own example output shows the visible difference directly: identical rule, identical traffic, but every logged line is prefixed with "Drop" instead of just the alert message, because the packet never reached its destination at all. That single word is the entire practical distinction between IDS and IPS mode, made visible in the log output itself rather than left as an abstract definition.

**Step 6: Pcap investigation mode**

Live traffic generation is useful for learning flags. Real investigations, most of the time, start with a pcap file someone already captured, handed off for analysis after the fact.

| Flag | Function |
|---|---|
| -r / --pcap-single= | Read a single pcap file |
| --pcap-list="" | Read multiple pcaps, space-separated within quotes |
| --pcap-show | Print which pcap is currently being processed |

Running the default configuration against mx-1.pcap with `sudo snort -c /etc/snort/snort.conf -A full -l . -r mx-1.pcap` generates 170 alerts. Continuing to read the same output, the Stream Statistics section shows 18 TCP segments queued, and the HTTP Inspect section shows 3 HTTP response headers extracted. Switching configuration files, from snort.conf to snortv2.conf, against the same mx-1.pcap drops the alert count to 68, the same pattern seen back in Step 2 where the two configuration files carry very different rule sets.

A second file, mx-2.pcap, run against the default configuration generates 340 alerts, with 82 TCP packets in the protocol breakdown. Combining mx-2.pcap and mx-3.pcap in a single run with `--pcap-list="mx-2.pcap mx-3.pcap"` pushes the total to 1020 alerts across both files together.

That combined-file number matters for a reason beyond the raw count. --pcap-list processes multiple files as one continuous stream without distinguishing which alert came from which file in the summary total. --pcap-show exists specifically to solve that ambiguity, printing the active filename as processing moves from one pcap to the next, which is the only way to attribute a specific alert back to a specific source file when multiple captures get analyzed together.

**Step 7: Snort rule structure**

Everything up to this point uses Snort's built-in rules. Writing custom ones is where the tool actually earns the customizable half of its reputation. A rule splits into two parts: a header (action, protocol, IPs, ports, direction) and options (everything in parentheses that refines the match).

Actions define what happens on a match: alert (generate an alert and log the packet), log (log only, no alert), drop (block and log, IPS mode), reject (block, log, and terminate the session, the most aggressive IPS response).

Protocol filtering in Snort 2 supports exactly four values: IP, TCP, UDP, ICMP. Application-layer protocols like FTP or HTTP are not directly filterable by name; detecting FTP traffic means filtering TCP on port 21 instead, then optionally adding content matching on top to confirm it is genuinely FTP traffic and not something else using the same port.

IP and port filtering support several patterns beyond a single address. A single IP looks like `192.168.1.56`. A CIDR range looks like `192.168.1.0/24`. Multiple ranges get bracketed together as `[192.168.1.0/24, 10.1.1.0/24]`. The negation operator excludes an address or range, written as `!192.168.1.0/24`. Ports follow the same logic: a single port is just the number, a range uses a colon such as `1:1024` or `1025:`, and multiple discrete ports get bracketed as `[21,23]`.

Direction uses only two operators: `->` for one-way source-to-destination flow, and `<>` for bidirectional flow. There is deliberately no reverse-only operator; a rule needing the opposite direction just swaps which side is written as source and which as destination.

Rule options split into three categories. General options carry msg (the one-line description shown on a match), sid (the rule's unique ID, scoped by range: below 100 reserved, 100 through 999,999 shipped with the build, 1,000,000 and above for user-created rules), reference (an optional link to a CVE or other external context), and rev (a revision counter with no automatic history, meaning it is entirely the analyst's responsibility to track what changed between revisions).

Payload options inspect the actual packet content. content matches a specific ASCII or hex pattern, case-sensitive by default, and can be used multiple times in one rule. nocase disables that case sensitivity. fast_pattern tells Snort which content match to prioritize first when a rule uses more than one, since Snort's default behavior of testing the largest content string first is not always the most efficient choice for a specific rule.

Non-payload options inspect packet metadata rather than content. id filters on the IP ID field. flags filters on TCP flags, where F means FIN, S means SYN, R means RST, P means PSH, A means ACK, and U means URG. dsize filters on payload size with min and max bounds. sameip flags packets where source and destination IP are identical, a pattern almost always indicative of spoofing or a misconfigured device rather than legitimate traffic.

This task's hands-on exercise puts every one of those pieces to work against task9.pcap. Writing a rule to match IP ID 35369, `alert icmp any any <> any any (id:35369; sid:1000001;)`, then running it against the pcap and reading the resulting alert reveals the request name of the matched packet: TIMESTAMP REQUEST, an ICMP message type used for clock synchronization between hosts, functionally similar to a ping but querying time rather than reachability.

Clearing the log, commenting out the previous rule, and filtering for the SYN flag specifically with `alert tcp any any <> any any (flags:S; sid:1000002;)` returns 1 matching packet, a single connection-initiation attempt in the entire capture. Switching to Push-Ack, the flag combination that marks a TCP segment carrying application data ready for immediate delivery, with `alert tcp any any <> any any (flags:PA; sid:1000003;)` returns 216 matching packets, the overwhelming majority of the capture's meaningful traffic, consistent with PA being the flag combination that carries actual payload rather than connection setup or teardown.

The final rule targets UDP specifically rather than IP traffic in general: filtering for UDP packets with matching source and destination IP, written as `alert udp any any <> any any (sameip; sid:1000004;)`, returns 7 matching packets once unassigned (0.0.0.0), broadcast (255.255.255.255), and multicast (224.0.0.0/4) addresses get manually excluded from the raw count, since those ranges can trigger a same-IP-style match without representing genuine spoofing. Scoping the rule to UDP specifically rather than all IP traffic matters here, since protocols like DHCP and SSDP discovery ride on UDP and are exactly the kind of traffic that produces those broadcast and unassigned-address false positives in the first place.

Finally, a scenario question rather than a hands-on one: after an analyst successfully modifies an existing rule, which option must change to reflect that. The answer is rev, the revision counter, and it is worth internalizing why that single field matters operationally. With no automatic version history built into Snort itself, rev is the only signal, visible directly in the rule text, that a rule has been touched since it was first written, and by how many times.

**Step 8: Snort2 operation logic**

The final task steps back from hands-on exercises to the architecture holding everything above together.

Snort processes a packet through five sequential components: the Packet Decoder, which collects and prepares raw packets, Pre-processors, which arrange and normalize packets ahead of detection, the Detection Engine, which applies the loaded ruleset, Logging and Alerting, which generates output on a match, and Output Plugins, which route that output to its final destination, whether a flat file, syslog, or a database.

Rules themselves come in three tiers. Community Rules are free, GPLv2-licensed, and publicly available with no registration. Registered Rules are also free but require registration, delivered with a 30-day delay behind the newest release. Subscriber Rules are paid, the same ruleset Registered users eventually get, but delivered in real time and updated twice weekly on Tuesdays and Thursdays. That 30-day gap between Registered and Subscriber access is a real, deliberate business choice: fresh coverage against genuinely new threats is the paid product's differentiator, not a technical limitation.

Configuration itself centers on a handful of variables inside snort.conf: HOME_NET, the network being protected, EXTERNAL_NET, everything else, typically set to !$HOME_NET, and a set of rule path variables, RULE_PATH, SO_RULE_PATH, and PREPROC_RULE_PATH, pointing at where different rule categories live on disk. Below that sits Data Acquisition module selection, which determines how Snort actually reads packets off the wire. PCAP is the default, passive sniffer mode. Afpacket is inline mode, what IPS blocking requires. Ipq and Nfq are inline modes specific to different Linux netfilter integrations. Ipfw handles inline on OpenBSD and FreeBSD via divert sockets. Dump is a testing mode for inline behavior without live enforcement. PCAP and Afpacket cover the overwhelming majority of real deployments; the rest exist for specific platform integrations most environments never touch.

## Why it matters

This room's lab runs Snort build 149, version 2.9.7.0, a build old enough to predate most of Snort's modern architecture. That matters for context, not just trivia. Snort 2 runs as multiple separate processes, one per CPU core, each requiring a full restart to reload policy changes; a crash in one process only affects that process, generating an isolated core dump rather than taking the whole system down. Snort 3, the actively developed version Cisco Talos now pushes new deployments toward, restructures this entirely into a single multi-threaded process with a shared backend control thread, trading Snort 2's crash isolation for faster reloads, lower memory and CPU overhead, and meaningfully better HTTP inspection performance. Cisco has already made Snort 3 the default inspection engine for new Firewall Threat Defense deployments, and legacy Snort 2 versions have been moving through scheduled end-of-life dates. Learning Snort 2's command syntax and rule language is not wasted effort even so. The rule language, the flag logic, and the operational thinking behind picking a detection technique all carry forward almost unchanged; what changes under Snort 3 is performance and architecture, not the conceptual model a working analyst needs to hold in their head.

The signature-versus-behavior tradeoff covered in Task 3 is not a theoretical distinction either. It is the daily reality of running any IDS/IPS in production. Signature-based detection, which is what the overwhelming majority of Snort rules in this room rely on, catches known threats with high precision and near-zero false positives, and catches nothing it has never seen a pattern for. That gap is exactly why NBA and other behavior-based approaches exist as a complement rather than a replacement, and why a security team running Snort in isolation, with no anomaly-based layer behind it, is only ever defended against yesterday's attacks.

The rule-writing exercises in Task 9 map directly onto what a detection engineer actually does day to day: take a known bad pattern, whether an IP ID from a specific attack tool, a TCP flag combination consistent with reconnaissance, or a same-IP anomaly consistent with spoofing, and encode it precisely enough to catch real instances without drowning the queue in false positives from adjacent, legitimate traffic. Scoping the fourth rule to UDP rather than all IP traffic is a small example of that same discipline in practice: a broader rule would have caught the same 7 packets here, but it would also have caught every other same-IP anomaly across TCP and ICMP traffic that had nothing to do with the pattern actually being hunted. The discipline of tracking rev on every change carries the same logic forward over time. A security team running dozens or hundreds of custom rules with no consistent revision tracking loses the ability to answer a basic incident-response question, which is exactly when a specific detection rule started or stopped working.

## Key takeaways

- IDS detects and alerts; IPS detects and actively blocks. The choice between them is a real tradeoff between the cost of a false positive breaking legitimate traffic and the cost of a missed detection doing nothing about it.
- NBA works like NIPS but requires a baselining period to learn normal traffic first, which is also its biggest weakness if an attacker is already active during that window.
- Snort's four sniffer-mode flags escalate from a one-line summary to a full hex-and-ASCII packet dump; picking the least detailed flag that answers the question keeps output readable.
- -K ASCII logging trades binary efficiency for direct human readability, organized into folders by source IP with connection-named files.
- Snort needs root privileges to sniff traffic, which means every log file it creates is root-owned and needs deliberate ownership handling to read back as a regular user.
- Only -A console and -A cmg print alerts live to the terminal; full, fast, and none all write silently to disk, or nowhere for none, and require reading the alert file afterward.
- Rule direction only supports -> and <>. There is no reverse-only operator; a reverse-direction rule just swaps which side is written as source.
- SID ranges are not arbitrary: below 100 is reserved, 100 to 999,999 ships with the build, and 1,000,000 and above belongs to user-created rules.
- Scoping a rule to the narrowest protocol that fits the pattern, UDP instead of all IP traffic for example, avoids false positives from unrelated traffic that happens to match the same non-payload condition.
- rev is the only built-in signal that a rule has changed since it was written, and Snort keeps no automatic history behind it, making manual revision discipline entirely the analyst's responsibility.
